- **estado:** ESTRUCTURA APROBADA (usuario, 2026-07-21: re-split 68.7a–d + poster separado). Pendiente: luz verde del usuario al diseño para empezar a codear (CLAUDE.md §8).
- **fecha:** 2026-07-21
- **fuentes:** PRD §13.5, migración `20260720000001` (`thumbnail_pct`), `mint-upload-url` (`requireSignedURLs=true`), `mint-video-url` (`sign_stream_hls_url`), `propiedades-y-video.md`, `step3.tsx`

## 0. TL;DR

Al terminar el procesamiento (video `ready`, post-webhook 68.5), el agente puede **elegir qué frame es la portada** de su video: 3 sugeridos automáticos (25/50/75 % de la duración) + un slider de ajuste fino. Se guarda **solo el porcentaje** (`thumbnail_pct`, columna ya existente). Cloudflare Stream renderiza los frames **on-the-fly** desde su endpoint de thumbnails (sin R2, sin subir imágenes) — pero como los videos son `requireSignedURLs=true`, **cada URL de thumbnail necesita un token JWT firmado**, que hoy **no está asignado a ninguna EF**. Ese es el trabajo real y crítico de esta subtarea. La UI en sí es liviana.

**Tres decisiones que este spec resuelve** (los 3 bloqueantes documentados en Taskmaster):
1. **DÓNDE vive** → sección "Portada" en el **editor de publicación** (`my-listings → editar`), activa solo cuando el video está `ready`. No en el wizard síncrono (ahí el video aún está `processing`).
2. **Firma de thumbnails** → nueva EF pequeña **`mint-thumbnail-url`** (CRÍTICA, TDD), owner-scoped por `cloudflare_uid`, reutiliza `sign_stream_hls_url` de `clients.ts`. Un token firma todos los `?time=` del mismo uid.
3. **Persistencia + render** → `UPDATE property_videos SET thumbnail_pct` directo con RLS (patrón #17.8, sin EF nueva). El feed/detalle renderizan la portada con `?time = pct% × duración`; para eso se persiste `duration_seconds` (columna nueva, poblada por el webhook 68.5).

## 1. Contexto y problema

### 1.1 El timing choca con upload-first
El wizard de publicación es **síncrono** y publica con el video todavía en `processing` (el binario se subió directo a Stream; `ready` llega después por webhook 68.5). Por lo tanto **la portada NO se puede elegir dentro del wizard**: los frames de Stream no existen hasta que termina el transcode. Cualquier UI de portada que bloquee el wizard es incorrecta por diseño.

**Consecuencia:** la portada tiene dos momentos:
- **Al publicar (processing):** se aplica una **portada por defecto** automática (50 %). Cero fricción, el feed nunca muestra un hueco.
- **Cuando el agente quiera (ready):** puede **refinar** la portada en el editor. Opcional.

### 1.2 El gap de URL firmada (lo crítico)
`mint-upload-url` crea los videos con `requireSignedURLs=true`. Eso significa que **todo** acceso al video — manifest HLS **y también los thumbnails** — exige un token JWT firmado (RS256, `sub=uid`). Hoy:
- `mint-video-url` firma el **manifest HLS** del feed (helper `sign_stream_hls_url`), pero **no** emite URLs de thumbnail.
- Nadie firma thumbnails. Sin token, el endpoint de Stream responde **401**.

El picker necesita previsualizar **muchos** frames (los 3 sugeridos + cada posición del slider). Detalle clave que simplifica todo: **un token firma el `uid`, no el `time`** → **una sola** llamada de firma devuelve un token con el que el cliente arma `?time=Ns&token=T` para **cualquier** frame de ese video. No hay que firmar por-frame.

### 1.3 Limpieza de la portada legacy (deuda a cerrar)
`step3.tsx:162` todavía invoca `generate_and_store_thumbnail(cap_video_id, cap_local_uri, cap_duration)` — el path **P7 legacy**: extrae un frame del archivo **local**, lo sube a un **bucket público de Supabase** y setea `property_videos.thumbnail_url`. Con upload-first + Stream ese path quedó **muerto y contradictorio** (el video ya no vive en Supabase; la portada debe salir de Stream). **68.7 debe eliminar esa llamada** y reemplazar el origen de la portada por el thumbnail firmado de Stream. Es cierre de deuda, no scope nuevo.

## 2. Alcance

### En alcance
- **EF `mint-thumbnail-url`** (CRÍTICA): dado un `cloudflare_uid` del propio agente, devuelve `{ baseUrl, token, expiresIn, durationSeconds }` para previsualizar/renderizar frames firmados.
- **Componente `ThumbnailPicker.tsx`** (no-crítico): 3 sugeridos (25/50/75 %) + slider de ajuste fino, en superficie de gestión (clara).
- **Persistencia**: `UPDATE property_videos SET thumbnail_pct = <0-100>` con RLS owner (sin EF).
- **Default automático al publicar**: `thumbnail_pct = 50` si el agente no elige (aplicado sin bloquear el wizard).
- **Integración de render de portada**: el feed/detalle usan la portada firmada de Stream a `thumbnail_pct` (via `mint-video-url` extendido).
- **Cleanup**: eliminar `generate_and_store_thumbnail` del flujo de publish (portada legacy P7).
- **Columna `duration_seconds`** (migración aditiva): la poblará el webhook 68.5; necesaria para convertir pct→segundos server-side.

### Fuera de alcance (no-goals)
- Subir imagen externa como portada (PRD §13.5 lo prohíbe explícitamente en MVP).
- Portadas animadas (GIF `?duration=`) — Stream lo soporta, pero no está en el PRD.
- Editar/reemplazar el video (eso es §13.6, otra subtarea).
- `pg_cron` / regeneración masiva de portadas.
- Push "elige tu portada" cuando el video queda ready (nudge opcional → backlog, no MVP).

## 3. Decisiones de diseño (resolución de bloqueantes)

### Decisión A — Dónde vive la UI  ✅ Editor de publicación (my-listings → editar)
El picker vive como una **sección "Portada"** dentro del wizard **en modo edición** (`step3.tsx`, donde hoy el picker de video está oculto en edit mode, #53). Regla de visibilidad:

| Estado del video linkeado | Qué se muestra en la sección Portada |
|---|---|
| `processing` (aún transcodificando) | Aviso "Tu video se está procesando. Podrás elegir la portada cuando termine." + preview del default 50 % en cuanto exista. |
| `ready` | Picker completo: 3 sugeridos + slider. |
| `failed` | No se muestra la sección (el error de video manda). |
| `archived` | Portada fija (miniatura ya no editable; el original vive en R2). |

**Por qué aquí y no un flujo nuevo:** reutiliza infra existente (`useLoadProperty`, `FormPrefiller`, edit mode por contexto). El agente publica → ve su listing → "Editar" → elige portada cuando el video esté listo. Cero pantallas nuevas en la navegación.

**Alternativa descartada (A2):** una pantalla dedicada post-publicación tipo "tu video está listo, elige portada". Más descubrible, pero es una pantalla nueva + un canal de notificación que no existe en MVP. Se puede sumar después como nudge sin retrabajo (la sección Portada del editor sigue siendo la fuente).

### Decisión B — Firma de thumbnails  ✅ EF `mint-thumbnail-url` (nueva, pequeña, crítica)
Nueva EF DI-pura, mismo patrón que las demás (`handler.ts` + `index.ts` + `types.ts` + `handler.test.ts`), reutiliza el firmador RS256 ya existente.

```
POST /functions/v1/mint-thumbnail-url
Auth: verify_jwt del gateway (JWT del agente)
Body: { cloudflare_uid: string }

200 → {
  baseUrl: "https://customer-<sub>.cloudflarestream.com/<uid>/thumbnails/thumbnail.jpg",
  token: "<JWT RS256, sub=uid, exp=now+ttl>",
  durationSeconds: number,     // de property_videos.duration_seconds
  expiresIn: number            // ttl en segundos (app_config.signed_url_ttl_seconds)
}
```

**Seguridad (fail-closed, misma disciplina que `mint-video-url`):**
- 🔒 El video debe pertenecer al agente del JWT: `property_videos.agent_id = caller` **o** el `properties.owner_user_id = caller` (video ya linkeado). Ajeno → **403 FORBIDDEN_NOT_OWNER**.
- 🔒 Solo videos `ready` (o `processing` para mostrar el default) tienen thumbnails; `failed`/inexistente → **404**.
- 🔒 Falta JWK/config de firma → **500**, nunca devuelve URL sin firmar.
- El cliente arma `?time=<Ns>&token=<token>` para cada frame. El token cubre todos los `time` del uid hasta `exp`.

**Por qué EF nueva y no extender `mint-video-url`:** distinto modelo de auth. `mint-video-url` es **público-por-propiedad-activa** (batch de `property_ids`, filtro `status='active'`); el picker es **owner-por-uid** sobre un video que puede estar en una propiedad aún no activa o recién linkeada. Mezclarlos ensuciaría el contrato de seguridad del feed. La EF nueva es ~40 líneas de handler.

### Decisión C — Persistencia y qué sirve Stream  ✅ Guardar solo pct; render con `?time`; no PATCH a Stream
- **Guardar:** `UPDATE property_videos SET thumbnail_pct = <0-100> WHERE cloudflare_uid = … ` con **RLS owner** (sin EF — patrón de edición directa #17.8). La columna y su CHECK (0-100) ya existen (migración `20260720000001`).
- **Render:** en todas partes la portada se construye como `thumbnail.jpg?time=<pct/100 × duration>s&token=<T>`. Nosotros controlamos el frame 100 %.
- **NO** se hace `PATCH` a `thumbnailTimestampPct` de Stream. `// ponytail:` todas nuestras referencias son firmadas y las armamos server/cliente con el `time` derivado del pct — el default de Stream nunca se muestra, así que sincronizarlo sería una llamada extra con un modo de falla nuevo y cero beneficio. Techo conocido: si algún día se usa una referencia sin firmar (p. ej. un embed público de Stream), habría que PATCHear; no es el caso en MVP.
- **`duration_seconds`:** el endpoint de thumbnails de Stream toma `time` en **segundos**, no en porcentaje. Guardar **pct** (robusto a re-encodes, es la columna existente) obliga a conocer la duración al renderizar. Se persiste una vez: **columna nueva `property_videos.duration_seconds numeric`**, poblada por el **webhook 68.5** desde el `duration` que Stream reporta al quedar ready. Migración aditiva idempotente + rollback + pgTAP.
  - *Alternativa descartada:* guardar segundos directos (rompería el CHECK 0-100 y el enunciado del PRD "guardar el timestamp como %"). *Alternativa descartada:* fetch de duración a Stream en cada render (una llamada de red por poster; caro).

## 4. Flujo de usuario (UX)

```
Publicar (wizard) ──► video 'processing' ──► publish_property_atomic linkea el video
                                              con thumbnail_pct = 50 (default aplicado)
                                                        │
                          [webhook 68.5: video 'ready', duration_seconds, thumbnail_url]
                                                        │
   my-listings ──► "Editar" ──► step3 sección "Portada" ──► (video ready) picker visible
                                                        │
        ┌───────────────────────────────────────────────┴───────────────────────────────┐
        │  3 sugeridos: [25%] [50%✓] [75%]      «¿Otro momento? → Ajustar»                │
        │                                                                                 │
        │  (Ajustar) ──► slider 0…100%, preview en vivo del frame ──► "Usar este frame"   │
        └───────────────────────────────────────────────┬───────────────────────────────┘
                                                        │
                         UPDATE property_videos.thumbnail_pct = <elegido>  (RLS owner)
                                                        │
                     feed/detalle renderizan la portada firmada a ese frame
```

Pasos:
1. El agente entra a **Editar** su publicación. Si el video sigue `processing`, la sección Portada muestra el aviso de "procesando" (no bloquea guardar los demás campos).
2. Cuando está `ready`, aparecen **3 miniaturas** (25/50/75 %). La activa (default 50 %) tiene check y borde Salvia.
3. Tocar una miniatura la selecciona (feedback inmediato). Se persiste al confirmar/guardar.
4. **"Ajustar"** despliega el **slider**: al arrastrar, un preview grande actualiza el frame (debounced) usando `?time` con el mismo token. "Usar este frame" fija el pct exacto.
5. Guardar → `UPDATE thumbnail_pct`. La portada nueva se ve en el feed/detalle en el próximo render (signed URL a ese frame).

`mint-video-url` (feed) extiende su salida por video con `posterUrl` firmado a `time = COALESCE(thumbnail_pct,50)/100 × duration_seconds`. Es un cambio CRÍTICO acotado sobre `make_video_url_minter`. ## 8. Edge cases

- **Video nunca llega a `ready`** (webhook no dispara / falla): la sección Portada se queda en "procesando". La portada default (50 %) no existe aún → el feed usa placeholder. No romper.
- **`duration_seconds` NULL** (webhook viejo / no pobló): render cae a un `time` fijo seguro (p. ej. 1s) o al thumbnail_url legacy si existe; el picker muestra "vista previa no disponible" pero permite guardar el pct igual (se aplica cuando haya duración).
- **Token expira mientras el picker está abierto** (>ttl): al fallar una imagen, re-mintar el token una vez y reintentar; si vuelve a fallar, mensaje fail-soft.
- **Agente ajeno intenta mintar** un uid que no es suyo → 403 (probado en RED).
- **pct fuera de rango** (bug cliente): el CHECK 0-100 en DB rechaza; el cliente clampa 0..100 antes de enviar.
- **Concurrencia:** dos ediciones de portada del mismo video → last-write-wins (aceptable; es un solo campo).
- **Archived:** video movido a R2 → `cloudflare_uid=NULL` → picker no aplica; portada fija.
